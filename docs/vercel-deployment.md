# Vercel Services deployment

The SERVIER Potion Lab can deploy as one Vercel project with two services:

| Service | Framework | Entrypoint | Route prefix |
|---|---|---|---|
| `web` | Next.js | `apps/web` | `/` |
| `api` | Express/NestJS | `apps/api` | `/api` |

`vercel.json` uses `experimentalServices`, so the Vercel project framework setting must be **Services**.

## Current demo deployment

Use the Git-backed Production alias for reviewer demos:

```text
https://servier-potion-lab.vercel.app
```

The alias points to the latest successful `main` deployment. Do not hand off immutable preview URLs from earlier deployments; they do not receive later UI, runtime, or environment fixes.

## Runtime configuration

| Variable | Scope | Purpose |
|---|---|---|
| `DATABASE_URL` | API/build | Prisma Postgres connection string. Required for migrations and API runtime. |
| `PRISMA_DATABASE_URL` | API/build | Prisma Postgres connection string variant created by the Prisma integration. |
| `POSTGRES_URL` | API/build | PostgreSQL connection string variant created by Vercel/Prisma integrations. |
| `NEXT_PUBLIC_API_URL` | Web | Generated by Vercel Services as `/api`; used by the browser client. |
| `NEXT_PUBLIC_API_BASE_URL` | Web | Optional override for non-Services deployments. |
| `API_PUBLIC_URL` | API/build | Public API URL used in OpenAPI server metadata. |
| `WEB_ORIGIN` | API | Optional CORS allowlist for cross-origin deployments. |
| `OPENAI_API_KEY` | API | Required by the real-model Potion assistant JSON and AI SDK streaming chat runtime. Store only in Vercel/project secrets. |
| `OPENAI_MODEL` | API | Required model slug for the Potion assistant and `create_potion` tool planner, for example `gpt-5.5`. |

## Database release behavior

The API build script runs `scripts/vercel-db-release.mjs` before TypeScript compilation.

- Outside Vercel, the script is a no-op, so local `pnpm build` and `pnpm verify` stay deterministic.
- On Vercel, the script requires `DATABASE_URL` and runs `pnpm --filter @servier-potion-lab/db db:migrate`.
- On preview deployments only (`VERCEL_ENV=preview`), it also runs `pnpm --filter @servier-potion-lab/db db:seed`.
- Production deploys do not auto-seed because seeding clears potion history and randomizes inventory.

## Prisma Postgres

The demo deployment uses Prisma Postgres from the Vercel Marketplace because the app already uses Prisma ORM and `prisma migrate deploy`. Configure the database variables for every environment you expect to build or run:

- Production needs database variables plus `OPENAI_API_KEY` and `OPENAI_MODEL`.
- Preview can use a separate database and may be seeded automatically during preview builds.
- Development can pull local values with `vercel env pull`, but do not commit generated `.vercel/.env*` files.

```bash
vercel integration add prisma/prisma-postgres \
  --plan free \
  -e preview \
  -m region=cdg1 \
  --name servier-potion-lab-prisma-preview \
  --no-env-pull
```

Use `--no-env-pull` for agent-driven setup so database secrets stay in Vercel and are not written to local files. The integration injects `DATABASE_URL`, which is enough for the existing Prisma migration and runtime flow.

## API service entrypoint

`apps/api/package.json` sets `main` to `dist/index.js` so Vercel's Express service uses the serverless Nest export. Local development still runs `dist/main.js` through the `dev` and `start` scripts.

## Commands

```bash
pnpm verify
vercel dev -L
vercel deploy
vercel deploy --prod
```

GitHub pushes to `main` trigger Production deployments after the Vercel Git integration is connected. Use direct `vercel deploy` commands only for intentional manual previews or redeploys.

## Smoke checks

After deploy:

1. Open the frontend root.
2. Check `GET /api/health`.
3. Check `GET /api/ready`.
4. Check `GET /api/docs-json`.
5. Create one potion from the UI and confirm inventory decrements plus recipes/ledger update.
6. Say `hi` in the Potion assistant and confirm text streams back, then ask for a known potion and confirm it returns a potion-created action card with approved-tool proof detail.

## Protected demo access

For short external demos, Vercel Deployment Protection can be bypassed with a temporary automation bypass URL. Treat that URL as a secret: do not commit it, paste it into docs, or expose it in CI logs. Revoke the temporary deployment-protection automation bypass after the handoff.
